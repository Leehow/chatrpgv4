# Director 引擎威胁机制自动驱动改造

## 目标
让 Story Director 自动驱动剧本定义的威胁机制——**threat clocks 自动推进**、**SAN 损失自动结算**、**danger attacks 在战斗场景自动发起对抗检定**——而不是只在玩家 stalled 时被动触发。当前这三者全是死的（clocks 恒为 0、SanitySession 从未被调用、dangers moves 纯文本）。

## 架构

新增 **`save/threat-state.json`** 持久化 clock 进度；给 **story-graph scene 加 `on_enter` 结构化钩子**（san_triggers / clock_ticks）；给 **threat-fronts danger 加可选 `attack_profiles`** 结构化对抗检定；在 **apply 层加场景进入钩子**（推进场景后 tick clocks、触发 SAN）；在 **driver 的 sanity_check 分支接入 SanitySession** 真正结算 SAN 损失。

数据流：`scene on_enter → director 生成 rules_requests(SAN/opposed) + pressure_moves(clock tick) → driver 执行(SanitySession/percentile) → apply 持久化(threat-state + sanity.json + events)`

## 新增/修改文件

**新建：**
- `plugins/coc-keeper/scripts/coc_threat_state.py` — threat-state.json 读写（load/save/tick_clock/check_full），封装 clock 持久化逻辑
- `tests/test_threat_engine.py` — clock tick、SAN 结算、danger attack 对抗检定测试

**修改（director 侧）：**
- `coc_story_director.py`：
  - `build_director_context`：读 threat-state.json 合并进 ctx 的 threat_fronts（clock 带 live current_segments）；读 scene 的 `on_enter` 钩子放入 ctx
  - `_build_rules_requests`：新增分支——scene on_enter 的 san_triggers → 发 sanity_check（带 san_loss_success/san_loss_fail_expr/source）；combat 场景的 danger attack_profiles → 发 opposed_check
  - `_build_pressure_moves`：新增 scene-on-enter clock tick（不再只依赖 PRESSURE/stalled）
- `coc_director_apply.py`：
  - 场景推进块（:481-499）后：新增"进入新场景钩子"——tick 关联 clocks、发 scene_enter 事件
  - pressure_moves 处理块（:380-407）：把 clock tick 写入 threat-state.json（tick 后检查 on_full，发 clock_full 事件）
  - sanity_check 结果：若 result 带 san_loss，调 SanitySession 结算并同步 investigator-state

**修改（driver 侧）：**
- `coc_playtest_driver.py:_execute_rules_requests`：sanity_check 分支改为加载 SanitySession → `sess.sanity_check(source, san_loss_success, san_loss_fail_expr, ...)` → `sess.save()`，返回含 san_loss/san_after 的 result

**修改（数据/schema）：**
- `story-graph-schema.md`：文档化 scene 的 `on_enter` 可选字段（san_triggers[]/clock_ticks[]）
- `coc_scenario_compile.py`：对 on_enter 做 warning 级校验（malformed → warn not error）
- `references/starter-scenarios/the-white-war/story-graph.json`：给关键场景加 on_enter（blast-chamber 的 SAN 0/1 violence、shelter 的 clock tick、dawn 的 danger attack）
- `references/starter-scenarios/the-white-war/threat-fronts.json`：给 dangers 加 attack_profiles（tentacle slash/wind blast 的结构化 skill_resist/damage）
- `references/state-schema.md`：文档化 threat-state.json
- `coc_state.py:_initialize_campaign_runtime_files`：初始化空的 threat-state.json

### Task 2: clock tick 接入 apply 层
改 `coc_director_apply.py` 的 pressure_moves 块：每个 tick move 调 `coc_threat_state.tick_clock` 持久化，检查 `current >= segments` → 发 clock_full 事件 + 返回 on_full 描述。改 `_build_pressure_moves` 支持 scene-on-enter 主动 tick。测试：跑一回合后 threat-state.json 有 clock 进度。

### Task 3: 场景进入钩子
改 `coc_director_apply.py` 场景推进块：推进到新场景后，读 scene 的 `on_enter.clock_ticks` → 生成 pressure_moves；读 `on_enter.san_triggers` → 注入 plan 的 rules_requests。发 `scene_enter` 事件。测试：推进到带 on_enter 的场景后 clocks 被 tick。

### Task 4: SAN 自动结算（接入 SanitySession）

### Task 5: director 发场景级 SAN request

### Task 6: dangers 结构化 attack_profiles
改 threat-fronts.json：给 dangers 加可选 `attack_profiles[]`（{name, attack_skill, attack_target_percent, resist_skill, damage, lethality, ignores_armor}）。保留 moves 文本给 LLM。改 `coc_story_director.py` 在 combat 场景从 attack_profiles 生成 opposed_check rules_requests。测试：dawn-counterstroke 场景产生对抗检定。

### Task 7: 给 White War 剧本填数据
给 story-graph.json 关键场景加 on_enter（blast-chamber san_triggers、shelter clock_ticks、blizzard/dawn danger 引用）；给 threat-fronts.json 的 dangers 加 attack_profiles（tentacle slash、wind blast、gale）。跑 compile 校验。

### Task 8: schema 文档 + 编译器 warning

### Task 9: 端到端集成测试
完整跑一局 White War，验证：clocks 真实推进并触发 on_full、SAN 损失结算到 investigator-state、danger 对抗检定执行、事件日志含 clock_tick/clock_full/san_check/scene_enter/attack 类型。

### Task 10: 双轨同步 + 全套回归

## 关键设计决策
- **clock 状态独立文件**（threat-state.json）而非写回 scenario/threat-fronts.json（后者是不可变剧本定义，runtime 状态不该污染）
- **director 只请求，driver/apply 结算**（遵守 director 文件头 "never modifies save/combat/sanity" 约束）
- **SAN 去重**：world-state 的 san_triggers_fired 防止重复扣 SAN
- **attack_profiles 可选**：不破坏现有 dangers（纯文本 moves 仍给 LLM Keeper）；只有填了 attack_profiles 的 danger 才被 director 自动驱动
- **向后兼容**：所有新字段可选；haunting 等现有剧本不加 on_enter 也能正常跑（行为不变）
